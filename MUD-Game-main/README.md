# 核心分析与迭代备忘录 - Sprint 1

## 1. 项目概述
**项目名称**：Java 文字冒险游戏 (Text Adventure / MUD)
**核心目标**：构建一个基于控制台（Console/Terminal）的纯文字交互游戏引擎，实现玩家在虚拟世界中的探索与战斗。

## 2. 技术架构状态与分析
### 核心组件
- **Game Loop (游戏循环)**：
    - 采用标准的 `while(running)` 循环模式。
    - 流程：`Input (读取指令)` -> `Process (解析与执行)` -> `Output (渲染结果)`。
    - **亮点**：支持从标准输入 (System.in) 或文本脚本流 (如 `test_script.txt`) 灵活读取指令。
- **Command Parser (指令解析器)**：
    - 简单的字符串分割与匹配（例如：`go north`, `look`, `attack goblin`）。
    - 当前采用 `Verb + Noun` 的固定格式，有效控制解析复杂度。
- **World Model (世界模型)**：
    - **Room (房间)**：包含描述、出口映射 (`Map<Direction, Room>`)、以及实体引用（如 `Monster`）。
    - **Player (玩家)**：包含当前位置与基础战斗属性。
    - **Monster (怪物)**：封装了生命值 (HP)、受击判定与概率闪避逻辑。

### 技术栈与边界约束
- **语言**：Java (JDK 17+ 推荐)
- **代码管理**：GitHub 及其配套的 Kanban 面板用于 Sprint 进度与 Issue 追踪。
- **依赖限制**：仅依赖标准库 (`java.util.*`, `java.io.*`)，无第三方框架。
- **系统边界**：严格遵循纯控制台无 GUI、无网络通信的物理约束。
- **数据持久化**：Sprint 1 数据仅保存在内存中，重启即重置。

## 3. 关键风险与重构规划 (面向 Sprint 2)
1.  **指令解析的局限性与“上帝类”倾向**：
    * *现状*：随着 `attack` 等指令的加入，`Game.java` 中的 `switch-case` 结构开始膨胀，且 `Player` 类承担了过多的控制台打印职责，违反单一职责原则 (SRP)。
    * *规划*：在后续重构中引入**命令模式 (Command Pattern)** 消除分支僵化，并考虑引入事件总线或 **View 层** 剥离底层输出逻辑。
2.  **回归测试成本高**：
    * *现状*：每次修改核心逻辑后，手动在控制台敲击指令测试极易遗漏边缘分支。
    * *解决*：已在本项目中落地自动化测试方案，引擎初始化时优先尝试读取同级目录下的 `test_script.txt` 跑通测试用例。

## 4. MVP 功能交付清单 (Sprint 1)
1.  **游戏初始化**：成功启动游戏并渲染初始场景描述。
2.  **场景探索**：支持基础导航指令 (`go [direction]`, `look`)，实现跨房间的安全移动。
3.  **基础战斗**：支持 `attack [target]` 指令，完整覆盖目标校验、命中/闪避判定、血量扣除及实体死亡移除的动态流程。
4.  **系统指令**：支持 `help` 唤出操作指南，`quit` 随时安全退出。
5.  **自动化测试**：支持通过文本脚本一键注入指令，自动化验证上述所有功能分支流转的正确性。
